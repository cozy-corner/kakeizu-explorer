# 設計: 日本史 家系図エクスプローラー

- 日付: 2026-06-14
- ステータス: ドラフト（ユーザーレビュー待ち）

## 1. コンセプト

日本の歴史上の人物の家系を、**人物から人物へ渡り歩いて探索**する Web サービス。

- 左ペイン: フォーカス中の人物を中心とした家系グラフ。
- 右ペイン: フォーカス中の人物の Wikipedia 記事。
- 目玉機能: **任意の 2 人の血縁/姻戚経路を可視化**（例: 「織田信長と徳川家康はどう繋がっているか」）。

## 2. 位置づけ・前提

- 個人の趣味・学習用。ただし**一般公開する**。
- 品質バーは趣味レベル、運用コストは**無料枠**を狙う。
- 学習テーマ: **Neo4j / Cypher**、グラフ描画。

## 3. アーキテクチャ

Next.js 一体型を Vercel にデプロイする。フロントとバックエンドの境界は REST/JSON 契約で分離し（疎結合寄り・運用は 1 つ）、Neo4j の認証情報は API Routes の内側に隠す。

```
[ブラウザ SPA]
  ├─ グラフ描画（Cytoscape.js）           …左ペイン
  ├─ Wikipedia 記事描画                    …右ペイン
  │    └─ ja.wikipedia REST API を直接 fetch（CORS 許可済みのためバック経由不要）
  └─ fetch ↓ (REST/JSON)
[Next.js API Routes (Vercel Serverless)]   …Neo4j 認証情報を隠す BFF
  └─ neo4j-driver（公式）↓ (neo4j+s://)
[Neo4j AuraDB Free]                        …人物=ノード, 親子/配偶者/兄弟=エッジ
```

設計上の決定理由:

- **Wikipedia はバックエンドを経由しない**。REST API（`https://ja.wikipedia.org/api/rest_v1/...`）が CORS 許可済みのため、ブラウザから直接取得できる。経由する理由（キャッシュ・SSR/SEO）は今回優先度が低い。
- **Neo4j は必ず API Routes 経由**。接続情報をブラウザに晒せないため。
- スタックは Next.js（TypeScript）一体型。Vercel への最小構成デプロイ・既存知見・JS グラフ描画エコシステムに噛み合う。

## 4. API（Next.js API Routes）

| エンドポイント | 役割 |
| --- | --- |
| `GET /api/person/:id/neighbors?hops=2` | エゴグラフ（周辺の親・子・配偶者・兄弟を数ホップ）を返す。左ペイン描画用。 |
| `GET /api/path?from=:id&to=:id` | 2 人の最短経路を返す。Cypher の `shortestPath` を使用。目玉機能用。 |
| `GET /api/search?q=` | 人物名検索（探索の入口）。 |

レスポンスは `{ nodes: [...], edges: [...] }` 形式の JSON 契約とし、フロントは外部 API と同様に fetch する。

## 5. データモデル（Neo4j）

- ノード: `(:Person { qid, label, birth, death, image, wikipediaTitle })`
  - `qid`: Wikidata の Q 番号（一意キー）。
  - `wikipediaTitle`: ja.wikipedia の記事タイトル（右ペイン取得用）。
- エッジ:
  - `(:Person)-[:PARENT_OF]->(:Person)`
  - `(:Person)-[:SPOUSE_OF]-(:Person)`
  - `(:Person)-[:SIBLING_OF]-(:Person)`
- 親子関係は Wikidata の P22（父）/P25（母）/P40（子）を **`PARENT_OF` 方向に正規化**し、重複を排除する。

## 6. データ取り込み（ETL / ビルド前バッチ）

- 出どころ: **Wikidata**（人物関係を構造化・API 取得できる実質唯一の源）。
- 対象人物の条件:
  - **家族関係（P22/P25/P40/P26/P3373）を 1 つ以上持つ** 人物（＝孤立人物は取り込まない。これが収容の生命線）、かつ
  - **日本国籍（P27 = Q17）または 出生地が日本（P19→P17=Q17）**。
  - 注: 当初は「歴史上の人物は P27 を持たないため出生地救済が必須」と想定していたが、**実測の結果これは外れている**。著名な歴史人物（織田信長 Q171411 / 徳川家康 Q171977 / 豊臣秀吉 Q187550 / 源頼朝 Q242800 / 平清盛 Q281833）はいずれも P27=日本 を持つ。**P27 だけでほぼ足り、現行の出生地定義での救済の正味は実測 754 人**（下表参照）。ただし救済条件を広げれば増える余地があり、未確定点（§12）として残す。
- 取得方法: SPARQL の分割取得、またはダンプ抽出 → Neo4j へ一括ロード。実装時に取得安定性で判断する。
- 各人物の `wikipediaTitle`・生没年・画像も同時取得する。
- 親子方向の正規化と重複排除を ETL 内で行う。

### 実測値（2026-06-14, Wikidata Query Service で直接カウント）

| 指標 | 実測 |
| --- | --- |
| A. P27=日本 ∧ 家族関係あり | 18,643 |
| B. 出生地=日本（P19→P17=Q17）∧ 家族関係あり | 12,369 |
| C. (A ∪ B) = **実取り込みコアノード** | **19,397** |
| 救済の正味（B かつ P27≠日本, 直接カウント） | 754 |
| D. コアの家族関係トリプル（父+母+子+配偶者+兄弟, 正規化前・方向重複あり） | 70,112 |
| 参考: 出生地=日本 全体（家族関係フィルタなし） | 145,406 |

→ コアは **約 1.9 万ノード / 正規化前 7.0 万トリプル**（親子の双方向重複を正規化すれば実エッジは概ね 4〜5 万）。

**AuraDB Free の上限に対する余裕**: 公表値には揺れがあり（FAQ・GraphAcademy は 200,000 ノード / 400,000 リレーション、一部製品ページは 50,000 / 175,000）、**正式な技術ドキュメント（neo4j.com/docs/aura）には数値の明記がない**。Neo4j 自身も「Aura コンソールで現在値を確認せよ」としている。→ **インスタンス作成時にコンソールの実値で確定すること**。ただし悲観側（50k/175k）で当てても、コア 19,397 ノード（上限の約 38%）/ 正規化後 4〜5 万エッジ（同 25〜29%）で収まる。**前提（無料枠に収まる）は両上限で成立**。なお「家族関係フィルタ」を外すとコアは即 14.5 万ノードに膨らみ悲観上限を超えるため、このフィルタは必須。

## 7. 画面・操作（探索モデル）

- **エゴ中心 + クリックで再中心化**: フォーカス人物を中央に、周辺数ホップを表示。ノードをクリックするとその人物を中心に再描画し、人物から人物へ渡り歩く。
- 右ペイン: フォーカス人物の Wikipedia 記事を表示。
- **経路探索モード**: 2 人を選択 → 経路をグラフ上にハイライト表示。

## 8. グラフ描画ライブラリ

- **Cytoscape.js** を採用。グラフ/ネットワーク描画専用で、経路ハイライト・各種レイアウトが標準装備。エゴグラフ・経路表示の双方に最適。
- 代替案: React Flow（React 流の凝った人物カードを作る場合。ただし自動レイアウトは別ライブラリが必要）。

## 9. エラーハンドリング

- 人物に Wikipedia 記事が無い → 右ペインは「記事なし」を表示。
- 経路が存在しない（非連結な 2 人）→「経路が見つかりません」。
- Neo4j / Wikipedia 取得失敗 → リトライ + フォールバック表示。

## 10. テスト

- ETL の正規化（親子方向・重複排除）のユニットテスト。
- API Routes のクエリ結果整形テスト。
- 経路探索の既知ペア（著名な親子など）での検証。

## 11. デプロイ

- Vercel（フロント + API Routes）。
- Neo4j AuraDB Free。インスタンス作成時に**コンソールでノード/リレーション上限の現在値を確認**する（公表値に揺れがあるため。§6 参照）。
- いずれも無料枠で運用する。

## 12. 未確定 / 後続で詰める点

- ETL を SPARQL 分割取得とダンプ抽出のどちらにするか（取得安定性で判断）。
- 出生地ベース救済の具体条件。現行の「P19→P17=Q17」では正味 +754 人にとどまるが、旧国名・藩など P17 が日本に張られていない出生地や、P131（行政区域内）を辿る定義に広げれば取りこぼしを減らせる。広げる必要性とコストを実装時に判断する。
- AuraDB Free の実際の上限値（コンソールで確認・確定する。§6 参照）。
